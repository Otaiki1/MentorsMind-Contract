ISSUE #255: resolve_dispute Passes JavaScript Number Instead of Integer
------------------------------------------------------------------------
Problem: normalizeSplitPercentage returned a JavaScript float (e.g., 75.50) 
which caused precision loss. Contract expects integer percentage 0-100.

Files Fixed:
✓ mentorminds-backend/src/services/escrow.service.ts
  - Changed resolveDispute signature from (escrowId, releaseToMentor: boolean) 
    to (escrowId, mentorPct: number)
  - Added validation: mentorPct must be integer 0-100
  - Changed nativeToScVal type from 'bool' to 'u32'
  - Added JSDoc explaining percentage format

✓ mentorminds-backend/src/services/escrow-dispute.service.ts
  - Changed interface parameter from splitPercentage to mentorPercentage
  - Added Math.round() to normalize float to integer
  - Added validation for 0-100 range
  - Updated JSDoc

✓ mentorminds-backend/src/services/disputes.service.ts
  - Changed resolveDispute call from (escrowId, true) to (escrowId, 100)
  - 100 = 100% to mentor (full release)

✓ mentorminds-backend/tests/escrow.service.test.ts
  - Updated test to use integer percentage (100 instead of true)
  - Added validation tests for float, negative, and out-of-range values

✓ mentorminds-backend/tests/escrow-dispute.service.test.ts
  - Changed test parameter from splitPercentage to mentorPercentage

Contract Signature (Rust):
  pub fn resolve_dispute(env: Env, escrow_id: u64, mentor_pct: u32)
  - mentor_pct: 0-100 integer (e.g., 75 = 75% to mentor, 25% to learner)


ISSUE #256: createEscrow Does Not Set Deadline
-----------------------------------------------
Problem: create_escrow was called without deadline parameter, meaning funds 
could be locked forever if platform goes offline.

Files Fixed:
✓ mentorminds-backend/src/services/escrow.service.ts
  - Added deadline: number to CreateEscrowInput interface
  - Added deadline validation (must be future Unix timestamp)
  - Added deadline parameter to contract call: nativeToScVal(input.deadline, {type: 'u64'})
  - Added JSDoc explaining deadline format

✓ mentorminds-backend/src/services/sorobanEscrow.service.ts
  - Added deadline parameter to createEscrow method signature
  - Added validation that deadline is in the future
  - Added JSDoc

✓ mentorminds-backend/src/examples/bookings-with-escrow-release.example.ts
  - Created complete example showing how to calculate deadline
  - Formula: scheduledEnd + 7 days grace period
  - Shows conversion to Unix timestamp (seconds)

Contract Signature (Rust):
  pub fn create_escrow(
    env: Env,
    mentor: Address,
    learner: Address,
    amount: i128,
    session_id: Symbol,
    token_address: Address,
    session_end_time: u64  // <-- This is the deadline parameter
  ) -> u64

Usage Example:
  const deadline = Math.floor((booking.scheduledEnd.getTime() + 7*24*60*60*1000) / 1000);
  await sorobanEscrow.createEscrow({ ..., deadline });


ISSUE #254: scheduleEscrowRelease Never Called
-----------------------------------------------
Problem: escrowReleaseQueue existed but was never invoked. Funds were released 
immediately instead of after 48-hour delay.

Files Created:
✓ mentorminds-backend/src/services/escrow-release-queue.service.ts
  - scheduleEscrowRelease(): Schedules release 48 hours after completion
  - cancelEscrowRelease(): Cancels scheduled release (for disputes)
  - InMemoryEscrowReleaseQueue: In-memory implementation for dev/testing
  - EscrowReleaseQueue interface for production (Bull/BullMQ)

✓ mentorminds-backend/tests/escrow-release-queue.service.test.ts
  - Tests for scheduling, cancellation, immediate execution
  - Tests for replacing existing jobs
  - Tests for pending job count

✓ mentorminds-backend/src/examples/bookings-with-escrow-release.example.ts
  - Complete BookingsService example showing integration
  - confirmBooking(): Creates escrow with deadline
  - completeBooking(): Schedules 48-hour delayed release
  - openDispute(): Cancels scheduled release

Integration Points:
1. BookingsService.completeBooking() should call:
   await scheduleEscrowRelease({
     escrowId, mentorId, learnerId,
     sessionCompletedAt: new Date()
   }, escrowReleaseQueue);

2. BookingsService.openDispute() should call:
   await cancelEscrowRelease(escrowId, escrowReleaseQueue);

3. Only release immediately if mentee explicitly approves


ISSUE #253: No Contract Address Verification
---------------------------------------------
Problem: SOROBAN_ESCROW_CONTRACT_ADDRESS read from env with no verification.
Misconfigured address could interact with wrong/malicious contract.

Files Fixed:
✓ mentorminds-backend/src/services/sorobanEscrow.service.ts
  - Enhanced verifyContractVersion() to check contract exists on-chain
  - Added contract address validation
  - Added WASM hash verification (if CONTRACT_EXPECTED_WASM_HASH is set)
  - Added getContractVerificationStatus() method
  - Returns: "verified" | "unverified" | "mismatch"

✓ mentorminds-backend/src/examples/health-check-with-contract-verification.example.ts
  - HealthCheckService with contract verification status
  - verifyContractOnStartup() function for app initialization
  - Express route handler example
  - Example health check responses
